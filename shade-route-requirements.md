# 日陰ルート提案 PoC 要件定義

## 目的
東京都ハッカソン用プロトタイプ。地上移動時に、出発地→目的地で日陰の多い経路を提案する。

## スコープ(個人PoC / スマホ実機で動けばOK)
- 対象デバイス: **スマートフォン(縦画面)前提**。PC向けの汎用Webレイアウトは不要
- 対象エリア: 大手町駅周辺の実在の街区(半径300m程度の小さい範囲)
- 日時: 固定の日付・時刻(例: 8月中旬 14:00)でシミュレーションできればよい。リアルタイム時刻連動は任意
- **建物・道路データは実データを使用する**(架空のモックデータは使わない)
- **GitHub Pagesで公開する**(静的ホスティングのみ。サーバーサイド処理は持たない)

## データソース(実データ)
- **建物の高さ・footprint**: PLATEAU 千代田区の3D都市モデル(建築物モデル、LOD1〜2.2、CityGML形式)
  - 公開先: G空間情報センター(https://www.geospatial.jp/ckan/dataset/plateau-13101-chiyoda-ku-2023 等、最新年度版を使用)
  - 商用利用も含め無償で利用可能
  - CityGMLはやや扱いにくいため、PLATEAU GIS ConverterやFME等でGeoJSON(building footprint + height属性)に変換してから使う
- **道路ネットワーク**: OpenStreetMapのOverpass APIで大手町周辺(半径300m程度)の歩行者道路(highway=footway/pedestrian/sidewalk等)を取得し、ノード/エッジのグラフとして利用

## 技術スタック(提案)
- 地図表示: Leaflet + OpenStreetMapタイル(軽量・実装が速い。タッチ操作でのピンチズーム/ドラッグに標準対応)
- データ前処理: PLATEAU CityGML→GeoJSON変換(PLATEAU GIS Converter等)、Overpass APIからの道路網取得は**ビルド前のローカル作業**として行い、取得済みJSON/GeoJSONを静的ファイルとしてリポジトリに含める(GitHub Pages上で実行時にOverpass APIを叩くと安定しないため、事前取得データを同梱する)
- 日影計算: `suncalc`(太陽高度・方位角を計算するJSライブラリ)で、建物footprint×実測高さ×太陽角度から簡易シャドウポリゴンを算出
- 経路探索: 道路ネットワーク上でダイクストラ/A*。エッジ重み = 距離 + 日向区間へのペナルティ(シャドウポリゴンと道路セグメントの交差判定で日陰/日向を判定)
- デプロイ: 静的ビルド成果物を `gh-pages` ブランチ or `docs/` フォルダとしてGitHub Pagesに公開(バンドルツールのbase pathをリポジトリ名に合わせて設定)

## 機能要件
1. 地図上に対象エリアの道路網を表示
2. 指定日時での建物の影を半透明ポリゴンで地図上に重ねて表示
3. 出発地・目的地をクリックまたは選択で指定
4. 「日陰優先ルート」と「最短距離ルート」を計算し、地図上に色分け表示(比較可能に)
5. 各ルートの総距離・日陰区間の割合を簡易表示

## UI
- **スマホ縦画面(375〜430px幅程度)を基準にレイアウト**。PC画面での見え方は考慮しない
- Leafletの地図をメイン領域いっぱいに表示し、上部 or 下部に固定のコントロールパネル(出発地・目的地入力、日時設定、ルート計算ボタン)
- タップ操作前提(地図タップで出発地/目的地を指定できるようにする)
- 凡例: 日陰ルート(青系)/最短ルート(赤系)/影(半透明グレー)

## Out of Scope(今回やらない)
- リアルタイム気象データ連動
- 複数日/時間帯の比較機能
- PC/タブレット向けのレスポンシブ対応
- LOD4等の高精度屋内建物モデルの使用(建物形状はLOD1〜2.2のfootprint+高さで十分とする)
- サーバーサイド処理・API(GitHub Pagesは静的ホスティングのみのため)

## 完了条件
- GitHub Pagesで公開したURLにスマホでアクセス → 対象エリアの地図上に影が表示される → 出発地・目的地を指定すると日陰優先ルート/最短ルートの両方が表示されること
