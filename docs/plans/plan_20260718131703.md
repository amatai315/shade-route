# PLAN

## Goal

`shade-route-requirements.md` の要件に従い、大手町駅周辺(半径300m)を対象とした「日陰ルート提案 PoC」Webアプリを実装する。Leaflet地図上に実際の建物影と道路網を表示し、タップで指定した出発地・目的地間の「日陰優先ルート」と「最短距離ルート」を計算・比較表示できるようにし、GitHub Pagesで静的公開できる状態に仕上げる。

## Current Findings

- プロジェクトは空(git未初期化)。`raw-data/chiyoda-buildings.zip` にPLATEAU千代田区の建築物CityGML一式(約1.9GB)が同梱されている。
- CityGMLは `udx/bldg/*.gml` に3次メッシュ単位で21ファイル分割。座標系は `EPSG:6697`(緯度・経度・標高の順)。
- 大手町駅(35.6862, 139.7671)周辺半径300mをカバーするのは以下2ファイルのみ:
  - `udx/bldg/53394611_bldg_6697_op.gml`(建物1022件)
  - `udx/bldg/53394621_bldg_6697_op.gml`(建物1323件)
- 高さ属性 `bldg:measuredHeight`(uom="m")は全建物に1件ずつ存在。
- footprint(建物外形)は `bldg:lod0FootPrint` が存在しないため、全建物に1件ずつ存在する `bldg:lod0RoofEdge` のポリゴンを採用する。
- zip全体を展開しなくても、Pythonの`zipfile`ストリーム読みで対象2ファイルのみ処理すれば十分(実測: 168MB/254MB展開サイズ、圧縮20MB/31MB)。
- Overpass API (`overpass-api.de`) への疎通を実地確認済み。`User-Agent`ヘッダ付きPOSTで200 OK、大手町周辺のhighwayデータ取得成功。

## Decisions

- ビルド構成: **Vite + TypeScript**(UIフレームワークなし、素のDOM操作)。理由: PoCのスコープに対して軽量で、依存が少なく、GitHub Pages向けのbase path設定も標準的な手順で対応できるため。
- 幾何演算ライブラリ: `@turf/turf` を使用(ポリゴン交差判定・距離計算・最近傍点探索など、影とルートの判定に必須)。
- GitHubリポジトリ名: `shade-route`(現フォルダ名と一致)。`vite.config.ts` の `base` は `'/shade-route/'` とする。
- デプロイ手段: `gh-pages` npmパッケージを使い、`npm run deploy` で `dist/` を `gh-pages` ブランチにpushする方式(requirements.md記載の「gh-pagesブランチ」方式に準拠)。
- 大手町駅の中心座標は 35.6862, 139.7671 を採用し、この点からの半径300mで建物・道路を抽出する。
- 日時は**現在時刻を基準にリアルタイム反映**する(要件の「リアルタイム時刻連動は任意」を採用する形)。ページ読み込み時に「今日の日付」+「現在時刻を1時間刻みに丸めた時刻」を既定値とし、影の再計算は連続更新ではなく1時間刻みの粒度で行う(重い再計算を避けるための簡略化)。
  - UIコントロール: 日付ピッカー(既定=当日)+ 時刻を1時間単位で選べるスライダー/セレクト(0〜23時、既定=現在時刻の時)+「現在時刻に戻す」ボタン、を控えめなコントロールパネルに設置し、手動での日時変更も可能にする。

## 補足決定事項(ユーザーからの計画修正フィードバック反映)

- 「固定の日付・時刻のみ」から「現在時刻を基準にしたリアルタイム反映(1時間刻み粒度)+手動設定可能」に変更。requirements.md の日時要件セクションにある「リアルタイム時刻連動は任意」を採用する形。

## Open Questions Resolved

1. 「フロントエンドのビルド構成」→ Vite + TypeScript(UIフレームワークなし)を採用(ユーザー選択A)。
2. 「GitHub Pages公開先のリポジトリ名」→ `shade-route`、`base: '/shade-route/'` を採用(ユーザー選択A)。

## Scope

**含む:**
- CityGMLから大手町駅周辺300mの建物footprint+高さを抽出するデータ変換スクリプト、および出力済み静的GeoJSON(`public/data/buildings.geojson`)
- Overpass APIから同エリアの歩行者道路網を取得し静的GeoJSONへ変換(`public/data/roads.geojson`)。取得は今回の実装作業中に一度だけ実行し、以後はビルド・実行時に外部APIを呼ばない
- Leaflet地図表示(道路網レイヤー、影ポリゴンレイヤー)
- suncalcによる太陽位置計算と、建物footprint×高さ×太陽角度からの簡易シャドウポリゴン生成
- 道路網からのルーティンググラフ構築、Dijkstra/A*による「最短距離ルート」「日陰優先ルート」の計算(エッジ重み=距離+日向区間ペナルティ)
- 地図タップによる出発地・目的地指定、日時設定、ルート計算ボタン、凡例、距離・日陰割合の結果表示を持つスマホ縦画面UI
- GitHub Pages向けビルド設定(`base` path)と `gh-pages` によるデプロイスクリプト
- README.md(ローカル起動手順・GitHub Pagesデプロイ手順、日本語)

**含まない(Out of Scope, requirements.md準拠):**
- リアルタイム気象・時刻連動
- 複数日時の比較機能
- PC/タブレット向けレスポンシブ対応
- LOD4等の高精度屋内モデル
- サーバーサイド処理・API

## Steps

1. **データ変換スクリプト作成・実行(建物)**: `scripts/extract-buildings.mjs`(または`.py`)を作成し、zip内の対象2 GMLファイルをストリームパースして大手町駅周辺300m以内の建物を抽出、`lod0RoofEdge`ポリゴン+`measuredHeight`から `public/data/buildings.geojson` を生成する。
2. **道路データ取得・変換**: Overpass APIへ大手町周辺300mの歩行者道路(`highway=footway/pedestrian/sidewalk/residential`等)クエリを実行し、結果を `public/data/roads.geojson` として保存するスクリプト(`scripts/fetch-roads.mjs`)を作成・一度だけ実行する。
3. **プロジェクト scaffold**: Vite + TypeScript プロジェクトを初期化し、`leaflet`, `suncalc`, `@turf/turf`, 型定義, `gh-pages` を依存関係に追加。ディレクトリ構成(`src/main.ts`, `src/map.ts`, `src/shadow.ts`, `src/graph.ts`, `src/route.ts`, `src/ui.ts`, `src/style.css`)を作成。
4. **地図・データ表示実装**: Leafletで大手町中心の地図を初期化し、`roads.geojson`・`buildings.geojson` を読み込んで道路網表示、建物footprintは(必要なら)薄く表示。
5. **影計算実装**: suncalcで指定日時(既定=現在時刻を1時間丸め)の太陽高度・方位角を算出し、建物ごとにfootprint+高さから半透明シャドウポリゴンを生成して地図に重畳表示。日時コントロール変更時に再計算する。
6. **ルーティング実装**: 道路GeoJSONからグラフ(ノード/エッジ)を構築し、タップ地点を最近傍エッジにスナップ。Dijkstraで「最短距離ルート」と「日陰優先ルート」(エッジと影ポリゴンの交差判定で重み付け)をそれぞれ計算し、色分け表示(青=日陰ルート、赤=最短ルート)。
7. **UI実装**: スマホ縦画面レイアウト(固定コントロールパネル: 出発地/目的地表示、日付ピッカー+時刻スライダー(1時間刻み、既定=現在時刻)+「現在時刻に戻す」ボタン、計算ボタン、凡例)、地図タップでの出発地・目的地指定、結果表示(総距離・日陰区間割合)。
8. **ビルド設定・デプロイ準備**: `vite.config.ts` に `base: '/shade-route/'` を設定、`package.json` に `deploy` スクリプト(`gh-pages -d dist`)を追加。
9. **README作成**: ローカルでの動かし方(`npm install`, `npm run dev`)とGitHub Pagesへのデプロイ手順(`npm run build` → `npm run deploy`、GitHub側でのPages設定)を日本語で記載。
10. **動作確認**: `npm run build` が成功すること、`npm run preview` でモバイル幅表示を確認し、地図タップ→出発地・目的地指定→ルート計算→2ルート表示・影表示・凡例・距離/日陰割合表示までの一連の動作を確認する。

## Validation

- `npm run build` がエラーなく完了すること(型チェック含む)
- `npm run preview` を起動し、ブラウザのモバイル幅(375〜430px)エミュレーションで以下を確認:
  - 地図に道路網・建物の影が表示される
  - 地図タップで出発地・目的地が指定できる
  - ルート計算ボタンで「日陰優先ルート」(青)と「最短距離ルート」(赤)が両方表示される
  - 各ルートの総距離・日陰区間割合が表示される
- 生成した `buildings.geojson` / `roads.geojson` が実データ由来であること(モックでないこと)をファイル内容で確認
- 実行時に外部APIへのリクエストが発生しないこと(ネットワークタブ等で確認)

## Risks

- CityGMLの`lod0RoofEdge`は建物によって形状が単純な矩形ではない場合があり、シャドウポリゴン生成(footprint押し出し)のロジックが複雑な形状で破綻する可能性がある → 単純な凸包近似などフォールバックを用意する。
- Overpass APIから取得できる道路網の密度・連結性によっては、タップ地点から目的地までの経路が見つからない/不自然になる可能性がある → 取得範囲を若干広め(300m+バッファ)にする。
- 太陽高度が低い/日没後の時刻設定では影が極端に長くなる、または存在しない(夜間)ケースがある → 妥当な範囲にUIの時刻選択を軽く制約するか、影なしの表示にフォールバックする。
- GitHub Pagesへの実際のリポジトリ作成・push・Pages有効化はユーザー側の操作(またはユーザー承認)が必要になる可能性がある → README に手順を明記し、実際のpush/デプロイ実行前にユーザーに確認する。
